# البرومبت المتقدم لبناء منصة FuelMap ليبيا (جاهز للاستخدام)

انسخ النص التالي كما هو لاستخدامه مع نموذج ذكاء اصطناعي لبناء النظام:

```text
أنت مهندس حلول أول (Principal AI + Full-Stack Architect) ومطلوب منك تصميم وتنفيذ منصة ويب ذكية لعرض توفر الوقود في ليبيا في الزمن شبه الحقيقي، اعتمادًا على بيانات جماعية (Crowdsourcing) وخوارزميات تحقق وتنبؤ بالذكاء الاصطناعي.

## 1) الهدف النهائي
بناء منصة تصبح المرجع الأول في ليبيا لمعرفة:
- توفر البنزين والديزل لكل محطة.
- مستوى الازدحام الحالي والمتوقع.
- أقرب محطة مناسبة للمستخدم.
- تنبيهات فورية مفيدة تقلل وقت الانتظار والازدحام.

## 2) النطاق (MVP ثم التوسع)
### MVP (إلزامي الآن)
- تطبيق ويب متجاوب (Mobile-first + Desktop).
- خريطة تفاعلية لليبيا تعرض المحطات وحالتها بالألوان.
- نظام تحديث جماعي لحالة الوقود والازدحام.
- نظام تحقق ذكي يعتمد على الثقة + الزمن + القرب الجغرافي.
- نظام نقاط/موثوقية للمستخدمين.
- إشعارات ذكية أساسية.
- لوحة تحكم أساسية للمشرف (Admin) ولمحطة الوقود.

### Phase 2 (بعد MVP)
- تطبيق Android/iOS عبر API الحالية.
- نموذج تنبؤ متقدم لنفاد الوقود والازدحام.
- كشف التلاعب المتقدم (أنماط احتيال + حسابات وهمية).
- دعم دول إضافية ومحطات شحن EV.

## 3) الوظائف الأساسية
### 3.1 الخريطة التفاعلية
- عرض جميع المحطات على خريطة ليبيا.
- ألوان الحالة:
  - أخضر: متوفر
  - أصفر: كمية قليلة
  - أحمر: غير متوفر
  - أسود: مغلقة
- عرض تفاصيل المحطة: نوع الوقود، آخر تحديث موثوق، الازدحام، ساعات العمل.
- زر "أقرب محطة" بناءً على موقع المستخدم.

### 3.2 التحديثات الجماعية
المستخدم يستطيع إرسال تحديث لحالة محطة:
- الوقود: متوفر / قليل / غير متوفر.
- الازدحام: لا يوجد / متوسط / شديد.

كل تحديث يجب أن يتضمن:
- user_id
- station_id
- timestamp
- geolocation (lat/lng)
- optional image evidence

### 3.3 نظام التحقق الذكي
صمّم خوارزمية Confidence Score للتحديث تعتمد على:
- عدد التأكيدات المتوافقة خلال نافذة زمنية.
- وزن موثوقية المرسل.
- حداثة التحديث (Decay over time).
- المسافة بين المستخدم والمحطة.
- توافق الصورة (إن وجدت) مع نص التحديث.

مخرج الخوارزمية:
- `verified_status`
- `confidence_score` من 0 إلى 1
- `needs_manual_review` (true/false)

### 3.4 تقييم المستخدمين (Trust Score)
- لكل مستخدم score ديناميكي من 0 إلى 100.
- يزيد عند توافق تحديثاته مع النتائج النهائية.
- ينخفض عند التحديثات الخاطئة أو المشبوهة.
- ينعكس مباشرة على وزن تحديثاته المستقبلية.

### 3.5 التنبؤ (AI Forecasting)
نموذج ML يتنبأ بـ:
- احتمال نفاد الوقود خلال الساعات القادمة.
- احتمال ازدحام شديد خلال نافذة زمنية.
- الوقت المتوقع لتحسن الحالة.

المدخلات:
- تاريخ تحديثات المحطة.
- أنماط يوم/ساعة.
- كثافة البلاغات.
- بيانات تشغيل المحطة (إن توفرت).

### 3.6 الإشعارات الذكية
إرسال إشعارات حسب تفضيلات المستخدم:
- توفر نوع وقود محدد قربه.
- تحذير ازدحام شديد.
- وصول شحنة جديدة.

فلاتر التخصيص:
- المدينة
- نصف القطر (المسافة)
- نوع الوقود

### 3.7 لوحة إدارة المحطة
تمكين حسابات المحطات من:
- تحديث التوفر رسميًا.
- إعلان وصول شحنة.
- تعديل ساعات العمل.
- نشر تنبيه خاص بالمحطة.

### 3.8 كشف التلاعب
بناء قواعد + نموذج شذوذ (Anomaly Detection) لكشف:
- تحديثات متكررة بشكل غير طبيعي من نفس النقطة.
- تضارب ممنهج من حسابات مترابطة.
- قفزات مفاجئة غير مبررة بالحالة.

### 3.9 التحليلات
لوحة Analytics تعرض:
- أكثر المحطات ازدحامًا.
- توزيع الطلب حسب المدن.
- أوقات الذروة اليومية/الأسبوعية.
- دقة التنبؤ والتحقق مع الزمن.

## 4) المكدس التقني الإلزامي
- Frontend: Next.js + React + Tailwind CSS
- Maps: Mapbox (مع دعم بديل Google Maps)
- Backend API: Node.js + Express
- Database: PostgreSQL + PostGIS
- AI Services: Python (FastAPI) + PyTorch/TensorFlow
- Vision: OpenCV + YOLO (عند توفر صور)
- Queue/Async: Redis + BullMQ
- Realtime: WebSocket / SSE
- Deployment: Docker + CI/CD

## 5) متطلبات غير وظيفية
- أداء: تحديثات شبه فورية (هدف < 5 ثواني لانتشار الحالة).
- التوسع: تصميم متعدد المناطق والمدن.
- الأمان: JWT + RBAC + rate limiting + audit logs.
- الخصوصية: تقليل بيانات الموقع وحفظها بسياسات واضحة.
- الاعتمادية: retries + dead-letter queues + monitoring.
- الملاحظة (Observability): logs/metrics/traces + health checks.
- دعم العربية RTL كامل في الواجهة.

## 6) نموذج بيانات مبدئي (Design First)
اقترح مخطط جداول واضح يتضمن على الأقل:
- users
- stations
- station_status_history
- crowd_reports
- trust_scores
- station_official_updates
- notification_preferences
- notifications
- fraud_flags
- model_predictions

مع:
- المفاتيح الأساسية/الخارجية
- الفهارس الضرورية
- أعمدة PostGIS للموقع
- سياسات retention للبيانات الزمنية

## 7) واجهات API المطلوبة
صمّم REST (أو REST + GraphQL) يتضمن:
- `POST /reports`
- `GET /stations`
- `GET /stations/:id`
- `GET /stations/nearby`
- `POST /stations/:id/official-update`
- `GET /predictions/:stationId`
- `GET /analytics/overview`
- `POST /notifications/subscribe`

لكل endpoint قدم:
- schema للطلب/الاستجابة
- validation rules
- أخطاء متوقعة
- صلاحيات الوصول

## 8) منطق التحقق (مطلوب شبه-كود)
اكتب pseudo-code واضح لدالة:
`compute_station_state(station_id)`
تقوم بـ:
1. جلب آخر التقارير في نافذة زمنية.
2. وزن كل تقرير بحسب Trust + Distance + Recency.
3. حساب الحالة المرجحة (fuel + congestion).
4. حساب confidence.
5. تحديث الحالة الحالية إذا تجاوز confidence العتبة.

## 9) منطق مكافحة التلاعب (مطلوب)
اكتب قواعد عملية مثل:
- Max reports/hour لكل مستخدم ولكل محطة.
- Reject التقارير من مسافة تتجاوز حدًا منطقيًا.
- Flag عند نمط تصويت جماعي مشبوه.
- خفض وزن الحسابات الجديدة مؤقتًا.

## 10) المطلوب منك كمخرجات (Output Contract)
أنت ملزم بإخراج النتائج بهذا الترتيب:
1. Architecture Diagram (وصفي نصيًا).
2. Database Schema (SQL مبدئي).
3. API Spec (جداول واضحة).
4. Core Verification Algorithm (pseudo-code).
5. ML Pipeline (training + inference + monitoring).
6. Security & Privacy Checklist.
7. MVP Delivery Plan على 4-6 سباقات (Sprints).
8. مخاطر المشروع + خطط التخفيف.
9. أمثلة JSON جاهزة (request/response).
10. خطة اختبار شاملة (unit/integration/e2e/load).

## 11) قيود التنفيذ
- ابدأ بـ MVP قابل للإطلاق خلال 8-12 أسبوع.
- أي ميزة متقدمة يجب أن تكون خلف Feature Flag.
- لا تفترض توفر بيانات حكومية دقيقة؛ صمم النظام لبيانات ناقصة.
- الأولوية للدقة العملية وسهولة الاستخدام في بيئة اتصال متذبذب.

## 12) معايير القبول
يُعتبر المشروع ناجحًا مبدئيًا إذا:
- المستخدم يجد أقرب محطة متاحة خلال أقل من 10 ثوان.
- توجد آلية تحقق تقلل البلاغات الكاذبة بوضوح.
- الحالة الموثوقة للمحطة تتحدث بشكل مستمر وقابل للتفسير.
- الإشعارات الشخصية تعمل وفق المدينة/المسافة/نوع الوقود.
- لوحة التحليلات تعرض مؤشرات تشغيلية مفهومة.

الآن ابدأ مباشرة بإخراج القسم رقم 1 ثم 2 ثم 3... حتى 10 بنفس الترتيب، دون حذف أي قسم.
```

## ملاحظة استخدام
